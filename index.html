AI Email Ticketing Service | Automated Support Pipeline

Intelligent
Support,
Automated.

A continuously running AI pipeline that fetches support emails from Gmail, enriches them with semantic knowledge-base search, classifies them via local Llama 3 with confidence scoring, and routes through a three-tier workflow: auto-resolve (≥0.9), team-lead approval (0.7–0.9), or escalation (<0.7) — sending friendly, SOP-backed resolution emails at every step.

01How it works
The Pipeline
01
Email Ingestion
Connects to Gmail via IMAP over SSL, authenticating with an app password from config.yaml. Searches for UNSEEN messages since today's date. Parses multipart MIME, handles HTML/plain text, and marks emails \Seen atomically after ticket insertion. Filters out system notification emails (matching "High Confidence Ticket Notification" subject) to prevent self-loops.
02
Ticket Generation
Generates a unique Ticket ID INC-2026-{1000–9999}. Inserts into SQL Server with status "New". Sends immediate acknowledgement email to sender with ticket reference. Dispatches async AI enrichment to thread pool — ingestion never blocks on AI inference.
03
Hybrid Knowledge Search
Searches knowledge_base/ for policy context. Primary: semantic embeddings via Ollama (nomic-embed-text) with cosine similarity, cached by file mtime. Fallback: keyword search on terms >4 chars. Documents cached in memory; re-embedded automatically when files change. Top-K and min-similarity configurable.
04
AI Categorization & Confidence
Sends ticket + SOP context to Llama 3 via Ollama with strict JSON schema. Extracts category, subcategory, summary, confidence (0.0–1.0). Category whitelist: Payroll/Salary, Finance/Invoice, Technology/VPN, HR/Leave. Two-attempt retry with num_ctx=8192. Fallback: regex extraction → keyword scan → content-based inference (normalized keyword scoring with dominance guard).
05
Team Lead Routing (Config-Based)
Looks up team lead from team_leads array in config.yaml (not DB). Each lead maps to one or more categories. Defaults to TL_GENERAL if unmatched. Writes category, subcategory, summary, assigned lead, and status="Processed" to DB.
06
Confidence-Driven Workflow
Three-tier automation by confidence: ≥0.9 → auto-resolve: sends solution email (AI-rewritten from SOP), notifies ops, marks Resolved. 0.7–0.9 → PendingApproval: generates solution, emails team lead for Approve/Reject. 0.7> → Escalated: emails team lead directly, no auto-solution. Team lead replies (approval/rejection/custom) are parsed and forwarded to employee.
07
Ticket Notes & Duplicate Detection
TicketNotes table tracks lifecycle: SystemNote (auto events), UserReply (sender replies), PendingApproval (staged solutions). Duplicate detection: parses subject for INC-2026-XXXX on inbound mail — replies attach to existing ticket as UserReply notes instead of creating new tickets. Approval/Escalation replies from team leads also detected via subject patterns.
02Core Capabilities
Features
01
Continuous Polling
Core
The pipeline runs indefinitely, checking for new emails in a tight loop. When no unread messages are found, it sleeps for a configurable interval (default 10 s) before polling again. IMAP IDLE is partially supported with a configurable timeout, falling back to standard polling. Press Ctrl+C to trigger a clean KeyboardInterrupt shutdown.
02
Local AI | No API Keys
AI
All inference runs locally via Ollama. No email content is sent to external services. The Llama 3 model classifies tickets into four categories (Payroll/Salary, Finance/Invoice, Technology/VPN, and HR/Leave) and generates a concise one-sentence summary. Confidence is returned as a float between 0 and 1, driving the automation threshold logic downstream.
03
Semantic Knowledge Base Search
AI
Corporate SOPs in knowledge_base/*.txt are embedded via Ollama (nomic-embed-text) and matched using cosine similarity. Results cached by file mtime — edits auto-re-embed on next search. Configurable top-K (default 2) and min-similarity (default 0.55). Keyword fallback for terms >4 chars ensures context is always retrieved. No vector DB required.
04
Three-Tier Confidence Automation
Logic
Confidence gates the entire workflow: ≥0.9 auto-resolves (solution email sent, ops notified, ticket Resolved). 0.7–0.9 triggers PendingApproval — solution generated but held for team lead Approve/Reject. 0.7 escalates directly to team lead with no auto-solution. Prevents low-quality resolutions while keeping high-confidence tickets fully automatic.
05
Team Lead Approval Workflow
Logic
When confidence is 0.7–0.9, the AI-generated solution is emailed to the assigned team lead with "Reply Approved to approve, Reply Reject to Reject." Lead's reply is parsed (stripping quoted history), decision extracted, and on Approve the solution is sent to employee; on Reject the lead's custom message is forwarded. Escalation replies (<0.7) are similarly parsed and forwarded.
06
AI-Rewritten Solution Emails
AI
On auto-resolve (≥0.9) or approval, a second Ollama call rewrites the raw SOP into a friendly, step-by-step employee-facing email: greeting, numbered actions (max 6), plain English, no internal jargon (SOP, policy, routing). Falls back to generic template if Ollama unavailable. Sender name extracted from email for personalization.
07
Concurrent Thread Pool
Concurrency
Email ingestion runs synchronously on the main thread while AI enrichment tasks are dispatched to a ThreadPoolExecutor. The default pool size of 4 allows multiple tickets to be enriched simultaneously. Each async task creates its own isolated database connection to avoid locking conflicts and shared state. Pool size is configurable via processing.thread_pool_size in config.yaml.
08
Ticket Notes & Lifecycle Tracking
Tracking
A dedicated TicketNotes table tracks the full lifecycle. Note types: UserReply (sender replies), SystemNote (automated events: solution sent, resolved, approved, escalated), PendingApproval (staged solution text), AgentComment (reserved for manual input). Each note stores author and timestamp. Auto-creates table if missing.
09
Duplicate & Self-Loop Detection
Safety
Three duplicate guards: (1) Inbound subject parsed for INC-2026-XXXX — replies become UserReply notes, not new tickets. (2) Team lead approval/escalation replies detected via subject patterns ("Approval Needed:", "Escalated Ticket:"). (3) Self-loop filter skips emails with subject "High Confidence Ticket Notification" (the system's own ops alerts).
10
Layered Error Handling
Resilience
Every external dependency (IMAP, SQL Server, Ollama, SMTP) wrapped in try/except with structured logging. JSON parsing: two-layer fallback (strip markdown fences → regex extraction for each field). DB connections closed in finally blocks. Category validation cross-checks AI output against keyword inference from raw ticket content, overriding on mismatch with confidence cap. No single failure cascades to full shutdown.
03System Components
Architecture
Layer 01
IMAP Email Ingestion
Uses Python's standard imaplib over SSL to connect to imap.gmail.com. Reads email credentials and app password from config. Searches for UNSEEN messages since today's date. The email standard library handles MIME parsing, including multipart messages with plain text and HTML parts. Marks emails as \Seen atomically after successful ticket insertion. Filters self-generated notification emails by subject pattern.
Layer 02
SQL Server Database
Connects via pyodbc to a local SQLEXPRESS instance. Connection string from config. Three tables: Tickets (main record), TeamLeadMapping (legacy, kept for backward compat), TicketNotes (note thread). Team lead routing now reads from team_leads in config.yaml. Each AI task creates its own DB connection for thread-safe concurrent writes.
Layer 03
Knowledge Base & Embeddings
Plain .txt files in knowledge_base/ (HR, Payroll, VPN, Finance). Primary retrieval: semantic embeddings via Ollama nomic-embed-text, cosine similarity, cached by mtime. Fallback: keyword search on terms >4 chars. No vector DB or external embedding service — fully self-contained. Top-K and similarity threshold configurable.
Layer 04
Ollama Llama 3 (Categorization)
ollama Python client sends structured prompts to local Llama 3. System message enforces JSON-only output with 4 fields: category (whitelist), subcategory, summary, confidence (0–1). num_ctx=8192 handles injected SOP context. Two-attempt retry. Fallback chain: regex extraction → keyword scan → content-based inference (normalized keyword scoring with dominance guard).
Layer 05
Confidence Router & Approval Engine
Three-tier routing by confidence: ≥0.9 → auto-resolve: generates AI-rewritten solution email (second Ollama call), sends to sender + ops alert, marks Resolved. 0.7–0.9 → PendingApproval: stages solution, emails team lead for Approve/Reject. 0.7 → Escalated: emails team lead directly. Lead replies parsed (quoted history stripped), decision extracted, custom messages forwarded to employee. All tracked in TicketNotes.
Layer 06
Thread Pool & SMTP
ThreadPoolExecutor (default 4 workers) keeps ingestion non-blocking. Each worker gets isolated DB connection. SMTP via smtplib with STARTTLS on 587 (Gmail app password). Sends: acknowledgement on ticket creation, solution emails (auto/approval), ops confidence alerts, team lead notifications (creation/approval/escalation). Config-driven team leads replace DB mapping.

05Stack
Technologies
Runtime
Python 3.x
Core language. The standard library covers most of the pipeline: imaplib, email, smtplib, json, re, logging, concurrent.futures, datetime, math. Only pyodbc, ollama, pyyaml need install.
AI Inference
Ollama + Llama 3
ollama Python client to local Llama 3. No API keys, no external data transfer. Handles categorization, summarization, confidence scoring, and solution email rewriting (second call). Runs with num_ctx=8192 for large SOP context.
Embeddings
Ollama + nomic-embed-text
Semantic KB search uses nomic-embed-text via Ollama embeddings endpoint. Cosine similarity in Python (stdlib math). Cached by file mtime — auto-re-embeds on file change. No vector DB required.
Database
SQL Server (SQLEXPRESS)
pyodbc with Trusted Connection. Tables: Tickets, TeamLeadMapping (legacy), TicketNotes. Per-task connections for thread-safe concurrent writes. Auto-creates TicketNotes table if missing.
Email
IMAP + SMTP (Gmail)
imaplib SSL (port 993) for reading. smtplib STARTTLS (587) for sending: acknowledgement, solution emails, ops alerts, team lead notifications (creation/approval/escalation). Gmail app passwords required.
Configuration
PyYAML (config.yaml)
All runtime config: IMAP/SMTP creds, DB connection, Ollama model + embedding model, thread pool size, KB folder, polling intervals, semantic search params (top_k, min_similarity), team_leads array, logging. Keep out of version control.
Concurrency
ThreadPoolExecutor
concurrent.futures.ThreadPoolExecutor (default 4 workers). Ingestion loop non-blocking — submits AI tasks, continues polling. Each worker gets isolated DB connection. Pool size tunable via processing.thread_pool_size.
06Quickstart
Get Running
01
Install Dependencies
Only three packages need to be installed. Everything else ships with Python's standard library.
pip install pyodbc ollama pyyaml
02
Configure config.yaml
Edit config.yaml with all settings. Required sections: imap (Gmail email + app password), smtp (same creds + notification_email), database (pyodbc connection string), ollama (model + embedding_model), processing (thread_pool_size, kb folder, semantic_top_k, semantic_min_similarity, polling intervals), team_leads (array of {id, name, email, categories[]}), logging (level, optional file).
imap.email: your@gmail.com
imap.app_password: xxxx xxxx xxxx xxxx
smtp.notification_email: ops@company.com
database.conn_str: "Driver={ODBC Driver 17...}"
ollama.embedding_model: nomic-embed-text
team_leads: [{id: TL001, name: "...", email: "...", categories: ["Payroll / Salary"]}, ...]
processing.semantic_top_k: 2
processing.semantic_min_similarity: 0.55
03
Populate the Knowledge Base
Add .txt files to knowledge_base/. One file per domain: HR/Leave, Payroll/Salary, Technology/VPN, Finance/Invoice. Semantic search uses embeddings (cached by mtime — edits auto-pick up). Keyword fallback covers gaps. More specific policy text = higher AI confidence.
04
Start Ollama with Required Models
Pull both Llama 3 (categorization) and nomic-embed-text (embeddings). The pipeline uses both automatically via the Ollama client.
ollama pull llama3
ollama pull nomic-embed-text
ollama serve
05
Launch the Pipeline
Start the pipeline. Runs continuously, poll interval configurable. Logs show: ticket creation, KB search matches, AI categorization with confidence, routing, email dispatches (ack, solution, approval request, escalation, ops alert). Ctrl+C for graceful shutdown.
python run_pipeline.py

AI Email Ticketing Service | Automated Support Pipeline

Intelligent
Support,
Automated.

A continuously running AI pipeline that fetches support emails from Gmail, classifies them with a local Llama 3 model, routes them to the right team lead, and sends knowledge-backed resolution emails.

01How it works
The Pipeline
01
Email Ingestion
Connects to Gmail via IMAP over SSL, authenticating with an app password sourced from config.yaml. Searches for UNSEEN messages received since the current date. Parses the full email, handling multipart MIME structures correctly before any further processing. Each email is marked as \Seen immediately after being read, preventing any duplicate processing across restarts or crashes.
02
Ticket Generation
A unique Ticket ID is generated in the format INC-2026-{1000–9999} using a random integer in the range. The ticket is immediately inserted into SQL Server with a status of "New". An async AI processing task is dispatched to the thread pool, so email ingestion never waits for AI inference to complete.
03
Knowledge Base Search
Before calling the AI model, the system searches the local knowledge_base/ directory for relevant corporate policy context. A two-stage hybrid approach is used: first, Ollama semantically extracts the topic from the ticket content and matches it against policy files. If semantic search yields no results, a keyword fallback matches terms longer than four characters against all documents so context is always retrieved.
04
AI Processing
The ticket content and retrieved knowledge base context are combined into a structured prompt, sent to Llama 3 via the local Ollama client. The system prompt strictly instructs the model to return only valid JSON. The response is parsed for four fields: category, subcategory, summary, and confidence (0.0–1.0). Robust JSON cleaning handles markdown code fences and malformed outputs, with regex-based fallback extraction if parsing fails entirely.
05
Team Lead Routing
The AI-predicted category is used to query the TeamLeadMapping table in SQL Server. The appropriate team lead identifier is retrieved and written to the ticket record. If no mapping is found for the predicted category, the system defaults to TL_GENERAL to ensure every ticket is always assigned. The ticket status is then updated from "New" to "Processed".
06
Notifications and Solution
When the confidence score meets or exceeds the 0.9 threshold, two emails are dispatched via Gmail SMTP with STARTTLS on port 587. First, a high-confidence alert is sent to the configured notification_email address. Second, a solution email incorporating extracted knowledge base content is sent directly to the original sender. Both emails are only triggered above the threshold to minimize noise from low-confidence classifications.
07
Ticket Notes and Replies
System notes are automatically written to the TicketNotes table after each key event: when a solution email is sent, and when the ticket is fully processed and resolved. When a user replies to a solution email, the reply is detected by parsing the subject for the original Ticket ID pattern. Instead of creating a new ticket, the reply body is stored as a UserReply note to preserve the conversation thread without generating duplicate tickets.
02Core Capabilities
Features
01
Continuous Polling
Core
The pipeline runs indefinitely, checking for new emails in a tight loop. When no unread messages are found, it sleeps for a configurable interval (default 10 s) before polling again. IMAP IDLE is partially supported with a configurable timeout, falling back to standard polling. Press Ctrl+C to trigger a clean KeyboardInterrupt shutdown.
02
Local AI | No API Keys
AI
All inference runs locally via Ollama. No email content is sent to external services. The Llama 3 model classifies tickets into four categories (Payroll/Salary, Finance/Invoice, Technology/VPN, and HR/Leave) and generates a concise one-sentence summary. Confidence is returned as a float between 0 and 1, driving the automation threshold logic downstream.
03
Hybrid Knowledge Base Search
AI
Corporate SOPs in knowledge_base/*.txt are retrieved in two stages. Ollama first extracts the semantic topic from the ticket and checks which policy documents mention it. If this yields nothing, keyword extraction (only terms longer than four characters) falls back to direct document matching. This two-stage approach avoids needing a vector database while still providing meaningful context for the AI model.
04
Confidence-Based Automation
Logic
The system behaves differently depending on how certain the AI is. When confidence reaches or exceeds 0.9, it acts immediately: notifying the operations team and sending the sender a solution email. Below that threshold, the ticket is still categorized and routed, but no automated email is sent. This prevents low-quality resolutions from being sent automatically.
05
Concurrent Thread Pool
Concurrency
Email ingestion runs synchronously on the main thread while AI enrichment tasks are dispatched to a ThreadPoolExecutor. The default pool size of 4 allows multiple tickets to be enriched simultaneously. Each async task creates its own isolated database connection to avoid locking conflicts and shared state. Pool size is configurable via processing.thread_pool_size in config.yaml.
06
Internal Ticket Notes
Tracking
A dedicated TicketNotes table tracks the full lifecycle of each ticket. Three note types are supported: UserReply for email thread responses from the sender, SystemNote for automated events such as "Solution sent" and "Ticket resolved", and AgentComment reserved for future manual agent input. Each note stores the author (email address or system identifier) and a precise timestamp.
07
Duplicate Detection
Safety
When a user replies to a solution email, the subject line is parsed for the Ticket ID pattern (INC-2026-XXXX). If found, no new ticket is created. Instead, the reply body is stored as a UserReply note against the original ticket. This prevents the inbox from generating duplicate tickets on every email thread reply, keeping the database clean.
08
Layered Error Handling
Resilience
Every external dependency (IMAP, SQL Server, Ollama, and SMTP) is wrapped in its own try/except block with structured logging at every key step. JSON parsing includes two-layer fallback: first strip markdown fences and re-parse; then use regex extraction to pull the four expected fields. Database connections are always closed in finally blocks. No single failure cascades into a full shutdown so the pipeline keeps running.
03System Components
Architecture
Layer 01
IMAP Email Ingestion
Uses Python's standard imaplib over SSL to connect to imap.gmail.com. Reads email credentials and app password from config. Searches for UNSEEN messages since today's date. The email standard library handles MIME parsing, including multipart messages with plain text and HTML parts. Marks emails as \Seen atomically after successful ticket insertion.
Layer 02
SQL Server Database
Connects via pyodbc to a local SQLEXPRESS instance. The connection string is read from config. Three tables handle all persistence: Tickets for the main record, TeamLeadMapping for routing logic, and TicketNotes for the note thread. Each AI processing task creates its own database connection to avoid cross-thread locking and ensure safe concurrent writes.
Layer 03
Knowledge Base
Plain .txt files in the knowledge_base/ directory contain corporate SOPs organized by domain: HR leave policy, payroll procedures, VPN and technology SOPs, and finance invoice workflows. No vector database or external embedding service is required. The hybrid semantic and keyword search keeps the system fully self-contained and portable.
Layer 04
Ollama Llama 3
The ollama Python client sends structured prompts to the locally-running Llama 3 model. The system message strictly instructs JSON-only output. The returned object is cleaned, parsed, and validated for the four expected fields. Fallback regex extraction targets each field individually if the full JSON parse fails, ensuring a category and confidence score are always extracted even from partially malformed model output.
Layer 05
SMTP Notifications
Gmail SMTP with STARTTLS on port 587. Two distinct email types are sent when confidence ≥ 0.9: an operations alert to the notification_email address configured in config.yaml, and a solution email to the original ticket sender. The solution email embeds relevant extracted content from the matched knowledge base files, giving the sender a concrete answer drawn from official policy.
Layer 06
Thread Pool Executor
Python's concurrent.futures.ThreadPoolExecutor handles all AI enrichment asynchronously. The main ingestion loop submits a task and immediately continues polling; it never blocks waiting for Ollama. The pool defaults to 4 workers but can be tuned via processing.thread_pool_size. Each worker gets its own database connection, preventing shared-state issues and ensuring safe parallel writes across all concurrent tasks.

05Stack
Technologies
Runtime
Python 3.x
Core language. The standard library covers most of the pipeline, including imaplib, email, smtplib, json, re, logging, concurrent.futures, datetime. Only three third-party packages need to be installed.
AI Inference
Ollama + Llama 3
The ollama Python client connects to your locally running model. No API key, no external data transfer. Llama 3 handles classification, summarization, and confidence scoring.
Database
SQL Server
pyodbc connects to a local SQLEXPRESS instance via Trusted Connection. Manages Tickets, TeamLeadMapping, and TicketNotes. Thread-safe via per-task connections.
Email
IMAP + SMTP
imaplib over SSL for reading Gmail. smtplib with STARTTLS on port 587 for sending notification and solution emails. Both use Gmail app passwords.
Configuration
PyYAML
Loads config.yaml at startup: credentials, connection strings, model name, thread pool size, polling intervals. Keep this file outside version control.
Concurrency
ThreadPoolExecutor
concurrent.futures.ThreadPoolExecutor keeps the ingestion loop non-blocking. Default pool of 4 workers processes multiple tickets simultaneously with isolated database connections.
06Quickstart
Get Running
01
Install Dependencies
Only three packages need to be installed. Everything else ships with Python's standard library and does not require a separate install step.
pip install pyodbc ollama pyyaml
02
Configure config.yaml
Edit config.yaml with your Gmail address and app password (2FA must be enabled on the account to generate an app password), your SQL Server connection string, the notification email address, and your preferred thread pool size and polling intervals.
imap.email: your@gmail.com
imap.app_password: xxxx xxxx xxxx xxxx
database.conn_str: "Driver={ODBC Driver 17...}"
smtp.notification_email: ops@yourcompany.com
processing.thread_pool_size: 4
03
Populate the Knowledge Base
Add plain .txt files to the knowledge_base/ directory. Each file should cover one policy domain: HR leave policies, payroll procedures, VPN procedures, finance invoice workflows. The hybrid search handles retrieval automatically. More specific policy text leads to better AI context and higher confidence scores.
04
Start Ollama with Llama 3
Ensure Ollama is running locally with the Llama 3 model available. The pipeline connects to the Ollama client automatically. No configuration required beyond ensuring the service is running.
ollama pull llama3
ollama serve
05
Launch the Pipeline
Start the pipeline. It will run continuously, processing emails as they arrive. Watch the logs for ticket creation, AI enrichment results, confidence scores, and email dispatch confirmations. Press Ctrl+C for a graceful shutdown when needed.
python run_pipeline.py
